**Status:** Approved design (ready for implementation plan)

**Owner:** dengqi
**Supersedes / extends:** [2026-04-28-sluice-design.md](./2026-04-28-sluice-design.md)

## 0. What this spec is

A **delta** on the V1 design. V1 concepts (Item, Source/Fetcher/Processor/Sink
protocols, Pydantic config, sqlite state, commit-after-success, four-tier LLM
fallback) are unchanged unless a section below explicitly says so.

V1.1 adds **13 features** across three themes:

- **Ops & polish** (Items 1-6): close the deferred items called out in V1
  spec + first-week-of-use ergonomics.
- **Pipeline-shape additions** (Items 7-9): one new fetcher-chain fallback
  policy (Item 7), one new processor implementation (`limit`, Item 8), one
  new processor type (`enrich`, Item 9).
- **Channel infra + new sinks** (Items 10-13): shared AST-based markdown
  converter (Item 10) consumed by Telegram (Item 11), Feishu (Item 12),
  Email (Item 13).

Order in this document = order in the implementation plan, so dependencies
flow top-to-bottom (e.g. the markdown AST module is built before the three
sinks that consume it).

## 1. Goals

1. Close every "deferred to v1.1" item in the V1 spec (`sluice gc`).
2. Make sluice usable for sub-daily pipelines (hourly news, weekly digests).
3. Eliminate the failure mode "fetcher chain failed → item silently dropped"
   when RSS feeds already ship usable content.
4. Give pipelines stage-level control over how many items reach the
   expensive LLM stage (`limit`), independent of pipeline-level hard caps.
5. Open up content enrichment as a first-class concept (`enrich` protocol),
   shipping HN-comments enrichment as the proof of concept.
6. Ship Telegram, Feishu, and Email as alternative output channels with a
   shared markdown→platform AST converter so adding a 4th channel is a one-
   adapter job.
7. Keep V1's contracts intact: commit-after-success, idempotency by
   `sink_emissions`, no schema break for existing pipelines.

## 2. Non-goals (V1.1)

- Not adding new **sources** in V1.1 — Telegram-as-source, IMAP, Mastodon,
  X-list deferred to V1.2/V2.
- Not building a subscriber-management UI for the email sink — TOML static
  recipient lists only (chosen over SQLite table for simplicity).
- Not introducing a vector store / RAG over `seen_items.summary`.
- Not changing the four-tier LLM fallback shape, the `Item` dataclass shape,
  or the five core protocols.
- `enrich` is a **new processor subtype**, not a new top-level protocol.
  Reuses the `Processor` interface.

## 3. Schema changes (consolidated)

All schema additions land as a **single migration** (`user_version 1 → 2`)
applied at startup. New tables / columns:

```sql
-- For Item #3: local attachment mirroring
CREATE TABLE attachment_mirror (
  url_hash           TEXT PRIMARY KEY,   -- sha256(canonical_url)
  url                TEXT NOT NULL,      -- original URL
  local_path         TEXT NOT NULL,      -- relative to [state] attachment_dir
  mime_type          TEXT,
  byte_size          INTEGER,
  fetched_at         TEXT NOT NULL,
  last_referenced_at TEXT NOT NULL,      -- updated on every cache hit; gc evicts stale
  pipeline_id        TEXT NOT NULL       -- pipeline that first fetched it (audit only;
                                         -- file is shared across pipelines via url_hash)
);
CREATE INDEX idx_attachment_pipeline    ON attachment_mirror(pipeline_id);
CREATE INDEX idx_attachment_referenced  ON attachment_mirror(last_referenced_at);

-- For Item #1: gc tracking (audit trail of what was deleted)
CREATE TABLE gc_log (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  ran_at        TEXT NOT NULL,
  table_name    TEXT NOT NULL,
  rows_deleted  INTEGER NOT NULL,
  criteria      TEXT NOT NULL            -- e.g. "older_than=90d, status in (resolved, dead_letter)"
);

-- For Items #11/12/13: per-attempt delivery audit for push-channel sinks.
-- sink_emissions still holds the LATEST external_id per (pipeline, run, sink).
-- sink_delivery_log appends one row per send attempt (incl. retries within
-- a run, incl. per-recipient for email best_effort), so operators can see
-- "this run sent 3 telegram messages on 2 retries".
CREATE TABLE sink_delivery_log (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  pipeline_id   TEXT NOT NULL,
  run_key       TEXT NOT NULL,
  sink_id       TEXT NOT NULL,
  sink_type     TEXT NOT NULL,
  attempt_at    TEXT NOT NULL,
  ordinal       INTEGER NOT NULL,        -- send sequence within (run_key, sink_id):
                                         -- 1, 2, 3, ... — for tg per-item / feishu
                                         -- per-card / email per-recipient
  message_kind  TEXT NOT NULL,           -- "brief" | "item" | "chunk" | "recipient"
                                         -- aids `sluice deliveries` table layout
  recipient     TEXT,                    -- for email; NULL for tg/feishu
  external_id   TEXT,                    -- nullable: NULL on failure
  status        TEXT NOT NULL,           -- "success" | "failed"
  error_class   TEXT,                    -- nullable
  error_msg     TEXT                     -- nullable
);
CREATE INDEX idx_delivery_run    ON sink_delivery_log(pipeline_id, run_key, sink_id);
CREATE INDEX idx_delivery_status ON sink_delivery_log(pipeline_id, status);

-- For Item #4: metrics export (already-derivable from run_log; no new table.
--   We add an index for the metrics query path):
CREATE INDEX idx_runlog_started ON run_log(started_at);
```

`url_cache` gains a **size cap** but no schema change — a startup eviction
sweep enforces `max_rows`.

## 4. Items

### Item 1 — `sluice gc` (deferred from V1)

**Goal:** Reclaim space from `failed_items` (resolved + dead-letter older
than threshold), `url_cache` (expired + over-cap), and `attachment_mirror`
(unreferenced + expired).

**CLI:**
```
sluice gc                                    # uses defaults from sluice.toml
sluice gc --older-than 90d                   # threshold for failed_items
sluice gc --pipeline ai_news                 # scope to one pipeline
sluice gc --tables failed_items,url_cache    # subset
sluice gc --dry-run                          # report counts only
```

**Defaults** (in `[gc]` block of `sluice.toml`):
```toml
[gc]
failed_items_older_than = "90d"   # delete resolved/dead_letter older than this
url_cache_max_rows = 50000        # LRU eviction by fetched_at when over cap
url_cache_keep_expired = "7d"     # also delete rows where expires_at < now - this
attachment_unreferenced_after = "30d"   # rows with last_referenced_at older than this
```

**Eviction order per run:**
1. `failed_items`: WHERE `status IN ('resolved', 'dead_letter') AND last_failed_at < cutoff`
2. `url_cache`: WHERE `expires_at < (now - keep_expired)`, then if still over `max_rows`,
   delete oldest by `fetched_at` until at cap.
3. `attachment_mirror`: WHERE `last_referenced_at < (now - attachment_unreferenced_after)`.
   Each deletion also unlinks the file at `local_path`. Stale-by-time is
   the only eviction signal — there is no reference counter, since pipelines
   may stop using a URL silently.
4. Write one row per affected table to `gc_log`.

**Auto-run policy:** Off by default. Operator can wire it as a Prefect deployment:
```toml
[gc.schedule]
enabled = true
cron = "0 3 * * *"   # 3am daily
```
When enabled, `sluice deploy` registers a `sluice-gc` Prefect deployment in
addition to pipeline deployments.

**Edge cases:**
- gc must be safe to run while a pipeline is mid-run. We rely on SQLite's
  WAL + the fact that gc only deletes rows older than the threshold (in-flight
  rows are always newer).
- File deletions for `attachment_mirror` happen **after** the DB DELETE
  commits. Orphan files (DB row gone, file still on disk) are reaped by a
  separate "orphan sweep" step at the end of each gc. The sweep walks the
  fixed `{yyyy}/{mm}/{xx}/` directory tree (max ~1.2k leaf dirs over
  100 years) using `os.scandir` for efficiency — never a full recursive
  glob — and deletes files with no matching `local_path` in
  `attachment_mirror`. With 50k attachments and the sharded layout, sweep
  is O(scandir on ~1.2k dirs + 50k stat calls) — runs in well under 30s.

**Tests:** Unit tests per cleaner; integration test that runs gc against a
seeded DB + on-disk attachments and asserts both DB rows and files removed;
contract test that gc with `--dry-run` modifies nothing.

### Item 2 — Hourly / sub-daily pipelines + `run_key_template`

**Goal:** Allow pipelines to run more often than once per day without
breaking idempotency or `sink_emissions` semantics.

**Config:**
```toml
[pipeline]
schedule = "0 * * * *"               # hourly
run_key_template = "{pipeline_id}/{run_date}/{run_hour}"   # was hard-coded in V1
timezone = "Asia/Shanghai"
```

**Available template variables** (all derived from the run's start time in
pipeline timezone):
| Var | Format | Example |
|---|---|---|
| `{pipeline_id}` | string | `ai_news` |
| `{run_date}` | ISO date | `2026-04-29` |
| `{run_hour}` | `HH` | `14` |
| `{run_minute}` | `MM` | `30` |
| `{run_iso}` | ISO datetime | `2026-04-29T14:00:00+08:00` |
| `{run_epoch}` | unix seconds | `1777442400` |

**Default**: `run_key_template = "{pipeline_id}/{run_date}"` (V1 behavior).

**`lookback_overlap` semantics (revised)**: Default formula changes from
`max(1h, window * 0.2)` to `max(min(1h, window * 0.5), window * 0.2)`. For
hourly pipelines (`window = "1h"`), this gives 30min overlap instead of 1h
(which would double-count every item).

**Validation:** at config-load time:
1. Template must contain `{pipeline_id}` (hard requirement — disambiguates
   sink_emissions across pipelines).
2. Schedule-vs-template uniqueness check: take the next two consecutive
   fire times of the cron via `croniter`, render the template against each.
   If they produce the **same** `run_key`, raise
   `ConfigError("run_key_template does not vary with schedule cadence;
   adjacent runs would collide in sink_emissions")`.
3. This catches ordinary cadence collisions (hourly without `{run_hour}`,
   30-min without `{run_minute}`, etc.).
4. **DST collisions** are NOT exhaustively caught: a fall-back DST hour
   that happens months from now can silently produce the same `run_key`
   if the template uses only `{run_date}`+`{run_hour}` in a DST-observing
   `timezone`. Operators in DST-observing zones who run sub-daily
   pipelines should include `{run_iso}` or `{run_epoch}` in the template
   for guaranteed uniqueness. Documented in template-variable table; not
   enforced (would require scanning a full year of cron fires per pipeline
   at every config load).

**Migration impact on existing pipelines:** None. Existing TOMLs without
`run_key_template` keep V1 behavior. Existing `sink_emissions` rows remain
valid because `run_key` strings do not need to match a template.

**Edge cases:**
- DST transitions: `run_iso` carries timezone; `run_hour` is wall-clock in
  pipeline timezone. Two runs may produce the same `run_hour` on
  fall-back day — operator's responsibility to choose UTC-anchored template
  if this matters.
- Cron parsing: use `croniter` (already a Prefect transitive dep).

**Tests:** Unit tests for template rendering, lookback formula, validation;
integration test of two consecutive hourly runs against the same Notion sink
in `upsert` mode → two distinct pages (because `run_key` differs).

### Item 3 — Local attachment mirroring

**Goal:** Download RSS enclosure images / media, re-host locally, rewrite
URLs **on items** so downstream renderers emit local paths. Solves Notion /
email blocking hot-linked images.

**Scope clarification (per review 2026-04-29):** This stage operates on
**items only** — it walks `item.attachments` and rewrites configured
`item.extras.*` fields (typically `extras.cover_image`). It does NOT
post-process rendered markdown. For markdown image references inside an
item field (e.g. `item.fulltext` containing `![](https://…)`), the operator
must use a `field_filter` regex_replace step or a Jinja `render` template
that consumes the rewritten `extras.cover_image`. Markdown post-processing
of rendered output is **deferred** (could become a `rewrite_markdown`
processor in V1.2 if real demand surfaces).

**Attachment dataclass extension** (additive, V1 contract preserved):
```python
@dataclass
class Attachment:
    url: str
    # CAUTION: when [state] attachment_url_prefix == "" and the
    # mirror_attachments stage has run, `url` holds a RELATIVE PATH
    # ("2026/04/ab/abc.jpg"), NOT a fetchable URL. Sinks that require a
    # resolvable URL (email body, Notion page cover) must consume
    # `local_path` together with `attachment_url_prefix` to construct the
    # final URL — or rely on `format_url()` below. The Notion sink's
    # markdown handoff to notionify is exempt: notionify's `image_base_dir`
    # resolves relative paths internally as LOCAL_FILE.
    local_path: str | None = None    # NEW: relative path under attachment_dir
                                     #   (set by mirror_attachments after mirror)
    mime_type: str | None = None
    length: int | None = None
```

**`format_url()` helper** (lives in `sluice/state/attachments.py`,
exported for sink use):
```python
def format_url(local_path: str, prefix: str) -> str:
    """Compose the public-facing URL/path for a mirrored attachment.

    prefix == ""               → return local_path  (relative path; for
                                 notionify LOCAL_FILE consumption)
    prefix.startswith("file://") → return f"{prefix.rstrip('/')}/{local_path}"
    prefix.startswith("http://") | startswith("https://")
                              → return f"{prefix.rstrip('/')}/{local_path}"
    other                       → ConfigError at load time (validated in
                                 [state] section parser).
    """
```

`mirror_attachments` invokes `format_url` to populate `Attachment.url` and
the rewritten `extras.*` dot-paths. Sinks that need to compute URLs
themselves (e.g. email constructing an `<img src>`) call `format_url` with
the same prefix.

`local_path` is the canonical handle for "the mirrored file"; `url` is the
materialized public form. Sinks may consume either.

**Config:**
```toml
[state]
attachment_dir = "./data/attachments"
# attachment_url_prefix MODE choice — three valid forms:
#   ""   (or omitted) = "local-only" mode: Attachment.url = relative path
#                       (e.g. "2026/04/ab/abc.jpg"). Best for Notion sink
#                       (notionify uploads as LOCAL_FILE).
#   "file://..."     = "file URI" mode: Attachment.url = "file://<prefix>/<path>".
#                       Useful for local debug tools; INCOMPATIBLE with
#                       email sink (load-time ConfigError).
#   "https://..."    = "CDN" mode: Attachment.url = "https://<prefix>/<path>".
#                       Required for email; works for Notion (page cover OK).
attachment_url_prefix = ""

[[stages]]
name = "mirror_images"                # required, like all V1 stages
type = "mirror_attachments"
mime_prefixes = ["image/"]            # only mirror images by default
max_bytes = 10_000_000                # skip larger; log warning
ttl = "180d"                          # informational; eviction is by last_referenced_at
on_failure = "skip"                   # "skip" | "fail" | "drop_attachment"
rewrite_fields = ["extras.cover_image"]  # list of dot-paths to rewrite on each item.
                                         # Empty list `[]` = ONLY walk item.attachments,
                                         # don't touch any extras field. Default `[]`.
```

**Behavior:**
- Walks `item.attachments` for each surviving item, plus each dot-path in
  `rewrite_fields` (empty list = attachments only).
- For each URL: lookup `attachment_mirror` by `url_hash`. Hit → set
  `Attachment.local_path = local_path`, UPDATE `last_referenced_at = now()`,
  reuse file. Miss → HTTP GET (with SSRF guard, shared httpx pool), atomic
  write (see below), insert row.
- After mirror succeeds, sets:
  - `Attachment.local_path = "{yyyy}/{mm}/{xx}/{hash}.{ext}"`
  - `Attachment.url = format_url(attachment_url_prefix, local_path)`
  - For each `rewrite_fields` dot-path on the item: replace value with
    `format_url(attachment_url_prefix, local_path)`.
- `format_url("", path)` returns `path` (relative); `format_url("file://x", path)`
  returns `"file://x/path"`; `format_url("https://x", path)` returns
  `"https://x/path"`. Prefix trailing `/` normalized at load time.
- Hash-based path means deduplicated storage across pipelines.

**Atomic file write (revised per review 2026-04-29):**
1. Stream HTTP response to a fresh per-writer temp path
   `attachment_dir/{yyyy}/{mm}/{url_hash[:2]}/.{url_hash}.tmp.{secrets.token_hex(4)}`
   so concurrent writers cannot collide on temp paths.
2. Attempt `INSERT OR IGNORE` on `attachment_mirror` keyed by `url_hash`
   inside a transaction; do NOT commit yet.
3. **Branch on insert result:**
   - **Insert won (rowcount = 1)**: this writer is canonical.
     a. Attempt `os.replace(tmp_path, final_path)` (atomic on POSIX and Windows).
     b. **If replace succeeds**: COMMIT the DB transaction.
        - **If COMMIT fails** (disk full mid-WAL flush, etc.): the file is
          on disk under `final_path` but no row exists. SELECT by url_hash
          to verify; if absent, log error + leave file for gc orphan
          sweep (§4.1) to reap. Raise to caller. Don't try to delete the
          file inline — another concurrent writer may have legitimately
          re-inserted by then.
     c. **If replace fails** (disk full, permissions, etc.): ROLLBACK the
        DB transaction (so the row never reaches durable state),
        `os.unlink(tmp_path)` if it still exists, raise to caller. No
        dangling row pointing to a non-existent file is ever committed.
   - **Insert lost (rowcount = 0)**: another writer beat us. ROLLBACK
     (insert was a no-op anyway); discard our temp file
     (`os.unlink(tmp_path)`); re-SELECT the existing row to get
     `local_path`. Three sub-branches:
     a. **Row present, file exists**: UPDATE `last_referenced_at = now()`,
        COMMIT, reuse the winner's file.
     b. **Row present, file missing on disk** (rare; manual deletion or
        gc orphan-sweep race): treat as full miss — fall through to a
        fresh fetch attempt of the entire mirror flow from step 1
        (limited to one retry per `mirror_attachments` invocation to
        prevent loops).
     c. **Row absent** (winner rolled back between our IGNORE and
        re-SELECT, or gc deleted it in the gap): treat as full miss —
        retry the entire flow from step 1 (also limited to one retry).
   - We do NOT assume our bytes match the winner's bytes (URL responses
     are not deterministic — same canonical URL can return different
     content under cache busts, A/B tests, time-of-day variation). The
     winner's file is the system-of-record.
4. **Crash safety**: if the process dies between steps 3.a and 3.b, the
   row is still uncommitted (per (2) we held the transaction open) — on
   process exit SQLite rolls back automatically. The orphan `final_path`
   file (if `os.replace` happened to succeed before crash) is reaped by
   the gc orphan sweep (§4.1).

**Mode-specific sink integration:**

**Notion sink (`attachment_url_prefix = ""` recommended):** `notionify` v3
auto-uploads images referenced in markdown via `client.py:_process_images`
(handles `LOCAL_FILE` / `EXTERNAL_URL` / `DATA_URI`). With prefix `""`,
sluice rewrites both `Attachment.url` and `extras.cover_image` to relative
paths like `2026/04/ab/abc.jpg`; the Notion sink passes `attachment_dir`
as `image_base_dir` to `NotionifyClient`, and notionify's LOCAL_FILE path
loads the image and uploads to Notion. With prefix `https://...`, notionify
takes the `EXTERNAL_URL` path (downloads + re-uploads).

For page **cover** (item-level `extras.cover_image`): Notion API accepts
only HTTP(S) URLs. If prefix is `""` or `file://`, sluice **skips cover**
(logged at info level). If prefix is `https://`, sluice passes the URL
through. Operators who want both Notion-uploaded inline images AND a Notion
page cover must either (a) use `https://` prefix, or (b) keep cover URLs
as the original (skip cover-rewrite by omitting `extras.cover_image` from
`rewrite_fields`).

**Email sink integration (per review 2026-04-29):** HTML email needs HTTPS
URLs. At config load time, if a pipeline has BOTH an Email sink AND a
`mirror_attachments` stage AND `attachment_url_prefix` does NOT start with
`http://` or `https://`, raise `ConfigError`:
`"email sink with mirror_attachments requires attachment_url_prefix to be
http(s):// (current: '<value>'); local/file:// paths cannot be loaded by
email clients."` CID-embedded inline images (`multipart/related`) is
intentionally NOT in V1.1 scope (deferred to V2).

**Edge cases:**
- Content-Length unknown: stream and abort if `max_bytes` exceeded mid-stream.
- HTTP redirect to a different host: SSRF guard re-runs on final URL.
- Concurrent pipelines mirroring same URL: see atomic-write contract above.

**Tests:** Unit (URL hash → path, prefix concatenation), integration
(httpx mock returning image bytes → file written + DB row), concurrency
(two simultaneous writers → single final file, both writers succeed),
failure modes (oversize, SSRF blocked, network timeout), config-validation
(email sink + file:// prefix → ConfigError at load time).

### Item 4 — Run metrics export

**Goal:** Expose `run_log` data as Prometheus `/metrics` and as a CLI report,
so operators can dashboard cost/latency over time.

**CLI:**
```
sluice stats                            # last 7 days, all pipelines
sluice stats ai_news                    # one pipeline
sluice stats --since 30d --format json  # JSON output for piping
```

Tabular output columns: `pipeline | runs | success% | avg_items_in | avg_items_out | total_llm_calls | total_cost_usd | p50_duration | p95_duration`.

**Prometheus exporter:** Uses `prometheus_client.start_http_server`
(stdlib `http.server`-backed; no FastAPI/Starlette dependency). Started by:
```
sluice metrics-server --host 0.0.0.0 --port 9090
```

Exposes:
| Metric | Type | Labels |
|---|---|---|
| `sluice_run_total` | counter | `pipeline_id`, `status` |
| `sluice_run_duration_seconds` | histogram | `pipeline_id` |
| `sluice_items_in_total` | counter | `pipeline_id` |
| `sluice_items_out_total` | counter | `pipeline_id` |
| `sluice_llm_calls_total` | counter | `pipeline_id` |
| `sluice_llm_cost_usd_total` | counter | `pipeline_id` |
| `sluice_failed_items_open` | gauge | `pipeline_id`, `status` (failed/dead_letter) |
| `sluice_url_cache_rows` | gauge | (no labels) |

**Implementation pattern (per review 2026-04-29):** use a **custom
Collector** (subclass of `prometheus_client.registry.Collector`), NOT the
high-level `Counter()` / `Gauge()` API. The custom collector's `collect()`
method runs once per scrape, queries the SQLite tables, and yields fresh
`CounterMetricFamily` / `GaugeMetricFamily` / `HistogramMetricFamily`
objects with `add_metric` calls per labelset.

```python
from prometheus_client.core import (
    CounterMetricFamily, GaugeMetricFamily, HistogramMetricFamily,
)

# Fixed bucket boundaries for sluice_run_duration_seconds (V1.1 contract;
# Prometheus requires stable buckets across scrapes for histogram math):
RUN_DURATION_BUCKETS = (1.0, 5.0, 10.0, 30.0, 60.0, 300.0, 900.0, 3600.0, float("inf"))

class SluiceCollector:
    def __init__(self, db_path: str):
        self._db_path = db_path

    def collect(self):
        with sqlite3.connect(self._db_path) as conn:
            # --- counter: runs by pipeline + status ---
            runs = CounterMetricFamily(
                "sluice_run_total",
                "Total pipeline runs by status",
                labels=["pipeline_id", "status"],
            )
            for pid, status, count in conn.execute(
                "SELECT pipeline_id, status, COUNT(*) FROM run_log "
                "GROUP BY pipeline_id, status"
            ):
                runs.add_metric([pid, status], count)
            yield runs

            # --- histogram: run duration ---
            # Compute cumulative bucket counts + sum from run_log per pipeline.
            durations = conn.execute(
                "SELECT pipeline_id, "
                "       (julianday(finished_at) - julianday(started_at)) * 86400 AS dur "
                "FROM run_log WHERE finished_at IS NOT NULL"
            ).fetchall()
            by_pid: dict[str, list[float]] = {}
            for pid, dur in durations:
                by_pid.setdefault(pid, []).append(dur)
            hist = HistogramMetricFamily(
                "sluice_run_duration_seconds",
                "Pipeline run duration",
                labels=["pipeline_id"],
                buckets=RUN_DURATION_BUCKETS,
            )
            for pid, durs in by_pid.items():
                cumulative = []
                for upper in RUN_DURATION_BUCKETS:
                    cumulative.append((str(upper), sum(1 for d in durs if d <= upper)))
                hist.add_metric([pid], buckets=cumulative, sum_value=sum(durs))
            yield hist

            # ... remaining counters/gauges follow the same shape:
            #     instantiate MetricFamily, loop add_metric, yield.
```
Registered via `REGISTRY.register(SluiceCollector(db_path))`. The fixed
`RUN_DURATION_BUCKETS` is part of the V1.1 contract — changing it later
would invalidate Grafana dashboards built on the histogram. The collector
holds no in-process counter state; DB is the source of truth, restart-safe.

The CLI `sluice stats` command computes p50/p95 via SQL percentile (or
sorted-list slicing) directly from `run_log` — independent of the
histogram, since percentiles from histograms are approximations and the
CLI prefers exact values.

**Edge cases:**
- Counter "going backwards" only happens if rows are deleted from
  `run_log` (we do not delete `run_log` in gc — confirmed).
- Stateless collector + per-scrape DB query is fine at typical Prometheus

### Item 5 — `url_cache` size cap + ttl tuning

**Goal:** Prevent `url_cache` from growing unbounded on long-lived
deployments.

**Config:**
```toml
[fetcher.cache]
enabled = true
ttl = "7d"                # default unchanged from V1
max_rows = 50000          # NEW: LRU evict by fetched_at when exceeded
```

**Behavior:**
- `UrlCacheStore` keeps an instance-level `_puts_since_check` counter.
  On every Nth `put()` (default `N = 256`), run `SELECT COUNT(*)`. If
  count > `max_rows * 1.1`, delete oldest `fetched_at` rows until at
  `max_rows`, reset counter. Deterministic 1-in-N (not random sampling)
  guarantees a check at most every N inserts regardless of luck.
- For a typical workload of ~500 inserts/run × 24 runs/day = 12k/day, this
  yields ~47 size checks/day. SQLite `COUNT(*)` on a 50k-row table is
  sub-millisecond.
- `sluice gc` always sweeps expired regardless of size and unconditionally
  enforces the size cap (full COUNT + eviction).

**Edge cases:** Worst-case row count between checks is `max_rows × 1.1 + N`
= 50000 × 1.1 + 256 = 55256 rows (≈10% slack header + N inserts before
next check). Overshoot above the cap is `≈5256` rows, each row ~5KB
markdown → ≈26MB transient overshoot. Next put-check or gc brings it back.
Acceptable.

**Tests:** Unit: eviction kicks in at `> max_rows * 1.1`, leaves exactly
`max_rows` rows; ttl deletion respects `expires_at` regardless of size.

### Item 7 — Fetcher chain `on_all_failed = "use_raw_summary"`

**Goal:** When the fetcher chain (trafilatura → firecrawl → jina) fails
entirely, fall back to the RSS feed's own `<description>` / `<content:encoded>`
(`item.raw_summary`) as `fulltext`. Many feeds (Atom feeds, modern RSS) ship
full content; current behavior drops the item.

**Config:**
```toml
[fetcher]
chain = ["trafilatura", "firecrawl", "jina_reader"]
min_chars = 500
on_all_failed = "use_raw_summary"   # NEW value, alongside "skip" | "continue_empty"
```

**Semantics:**
- If all chain entries fail or return < `min_chars`, AND `item.raw_summary`
  is non-empty after strip, set `item.fulltext = item.raw_summary` and
  continue.
- If `raw_summary` is also empty/None, behave as `skip` (record to
  `failed_items`).
- The `min_chars` check **does not** re-apply to `raw_summary`. Owner
  decision (2026-04-29): "先保证全面性" — short raw_summary still goes through;
  downstream LLM stage handles "信息不足" itself.
- The fetcher cache is **not** populated with raw_summary fallbacks
  (only real fetcher successes go into `url_cache`).
- Cache hits bypass the fetcher chain entirely — `on_all_failed` only
  applies on cache miss + all chain entries failing. A cached successful
  extraction continues to be reused even if the upstream fetcher would
  now fail.

**Failure record (when raw_summary also empty):**
`failed_items.error_class = "FetcherChainExhaustedNoFallback"`,
`error_msg = "all fetchers failed and raw_summary unavailable"`.

**Tests:** unit test with a fake all-failing chain + non-empty raw_summary →
fulltext populated; same with empty raw_summary → failed_items row; assert
`url_cache` not written.

### Item 8 — `limit` stage processor

**Goal:** Stage-level item-count cap with sort + grouping. Distinct from
V1's `max_items_per_run` (pipeline-level hard cap, FIFO drop). Typical use:
`filter` cheap → `llm_stage rate` (1¢ per item) → `limit top 10 by relevance`
→ `llm_stage summarize` (10¢ per item) → `render`.

**Config:**
```toml
[[stages]]
name = "top_relevant"                 # required, like all V1 stages
type = "limit"
top_n = 10
sort_by = "extras.relevance"          # dot-path; required
sort_order = "desc"                   # "desc" | "asc"; default "desc"
sort_missing = "last"                 # "first" | "last" | "drop"; default "last"
                                      #   how to order items where sort_by is None

# Optional grouping (per owner decision: ship in V1.1, not deferred):
group_by = "source_id"                # dot-path; optional
per_group_max = 3                     # required if group_by set
```

**Algorithm (when `group_by` set):**
1. Bucket items by `item.get(group_by)`.
2. Within each bucket: sort by `sort_by` per `sort_order` / `sort_missing`;
   take the top `per_group_max` rows of that bucket.
3. **Flatten** all per-bucket survivors into one list.
4. Globally re-sort that flat list by `sort_by` per `sort_order` /
   `sort_missing`; take the top `top_n`.

**Algorithm (when `group_by` unset):** sort all surviving items by
`sort_by` per `sort_order`/`sort_missing`; take top `top_n`.

**Drop semantics (both algorithms):** Items dropped by `limit` are NOT
recorded in `failed_items` (they are not failures, just deselected). They
are NOT marked seen (V1 commit-after-success semantics: only items
reaching all sinks become seen). **Exception:** dropped items are recorded
in a per-run debug log if `[pipeline] log_dropped_items = true`, for
tuning.

**Validation:** if `sort_by` field is not present on any surviving item
when stage runs, log warning; behavior follows `sort_missing`. If
`sort_missing = "drop"` and all items lack the field → empty output (which
downstream stages must tolerate, same as V1 empty-pipeline rules).

**Numeric coercion:** if `sort_by` resolves to a string that looks numeric
(e.g. `"0.85"` from a non-`json`-parsed `llm_stage`), `limit` does NOT
coerce. Operator must use `output_parser = "json"` upstream — same rule as
V1's `filter` numeric ops.

**Tests:** sort desc/asc, missing first/last/drop, grouping with
per_group_max, top_n trim after grouping, empty input edge case, string-
sort_by + numeric-coercion-rejection.

### Item 9 — `enrich` processor type + `hn_comments` enricher

**Goal:** A new processor subtype that reads from a third-party source to
populate a field on existing items, without touching `fulltext` or replacing
the fetcher chain.

#### 9a. The `enrich` processor

A new processor `type = "enrich"`. Like `fetcher_apply`, it walks items,
runs an `Enricher` strategy, writes to a configurable output field. Unlike
`fetcher_apply`, it can be repeated (different enrichers, different fields).

**Enricher protocol** (new, lives in `sluice/core/protocols.py`):
```python
class Enricher(Protocol):
    name: str

    async def enrich(self, item: Item) -> str | None:
        """Return enrichment string (markdown), or None if not applicable.
           Raise on hard failure; processor decides skip/fail per config."""
```

Registered via `@register_enricher("hn_comments")`, lives in `sluice/enrichers/`.

**Stage config:**
```toml
[[stages]]
name = "hn_thread"                     # required, like all V1 stages
type = "enrich"
enricher = "hn_comments"
output_field = "extras.comments"       # dot-path; same resolver as llm_stage
on_failure = "skip"                    # "skip" | "fail"
cache = true                           # use url_cache (separate cache key namespace)
max_chars = 8000                       # truncate output (head_tail strategy)
concurrency = 4                        # per-enrich-stage worker pool

# Enricher-specific config goes under `config = { ... }`. Fields here are
# passed verbatim to the Enricher constructor; schema is defined per
# enricher class via Pydantic. Keeps the generic `enrich` stage shape
# stable while enrichers add their own knobs.
[stages.config]
url_pattern = "news\\.ycombinator\\.com/item\\?id=(\\d+)"   # default
base_url = "https://www.hckrnws.com"                         # default
request_delay_seconds = 10                                    # per-host delay (default 10)
top_comments = 20                                             # default
```

`on_failure = "skip"` records to `failed_items` with
`stage = "enrich:<enricher_name>"`, `error_class = <exception class>`. The
item itself stays in the pipeline with the field unset (downstream filters
on `not_exists` can drop it if desired).

#### 9b. `hn_comments` enricher

**Strategy:**
1. Match `item.url` against `news.ycombinator.com/item\?id=(\d+)` (default;
   configurable `url_pattern` per-stage).
2. If no match → return `None` (skip, no error).
3. If match → fetch `https://www.hckrnws.com/stories/{item_id}` (HTML).
4. Run a **dedicated parser** (`sluice/enrichers/hn_parser.py`) over the
   response → markdown rendering of the comment thread (top N=20 comments,
   nested with `>` blockquotes, author + score line).
5. Truncate to `max_chars` (head_tail strategy).
6. Return markdown.

**hckrnws.com contract risk:** This is a third-party site we do not control.
Mitigation:
- Parser unit tests pin against a frozen sample HTML in `tests/fixtures/hn/`.
- On parse failure (unexpected HTML structure), enricher raises
  `EnricherParseError` → respects `on_failure` policy.
- Configurable base URL: `[stages.config] base_url = "https://www.hckrnws.com"`
  so a future fork / mirror can be swapped without code change.

**Cache key namespace:** Cache hit/miss uses `sha256("hn_comments:" + url)`
to avoid colliding with fetcher cache for the same URL.

**Edge cases:**
- HN "Ask HN" / "Show HN" entries also live at `news.ycombinator.com/item?id=`
  but have no external article — the V1 rss source already passes them
  through. `hn_comments` enricher works the same way.
- Concurrency: shared httpx client (the same one already used by
  `jina_reader` etc.); per-stage semaphore enforces `concurrency`.
- Politeness (per review 2026-04-29): a **per-stage token bucket keyed by
  URL host** (`asyncio.Lock` + `last_request_at: dict[str, float]`)
  enforces `stages.config.request_delay_seconds` (default 10s) **across
  all `concurrency` workers within this stage instance**. With
  `concurrency = 4` and `delay = 10s`, this gives ~6 requests/min per host
  (not 24/min), because the bucket serializes per-host access across the
  stage's workers.
- **Multi-stage scope caveat:** Two `enrich` stages of the same enricher
  hitting the same host get **independent** buckets (one per stage
  instance). With two stages each `concurrency = 4` and `delay = 10s`,
  effective rate is ~12 req/min per host. Operators wanting a global
  cap should use a single stage. Documented in the stage description;
  not enforced (cross-stage rate limiting was rejected as over-engineering
  for V1.1).
- Best-effort politeness only. V1.1 does NOT parse `robots.txt` (deferred).

**Tests:** Unit (URL pattern match, parser against fixture HTML, truncation);
integration (httpx mock → cache hit on second call); contract (registered
via decorator, smoke-tested by sample TOML).

### Item 10 — Markdown AST converter (shared infra for sinks 11/12/13)

**Goal:** Single source of markdown parsing for Telegram, Feishu, and Email
sinks. Each sink implements a renderer over a common token stream.

**Module:** `sluice/sinks/_markdown_ast.py`
**Library:** `markdown-it-py` (CommonMark spec, stable token stream API,
already a transitive dep of many python tools).

**Public API:**
```python
def parse_markdown(md: str) -> list[Token]:
    """Parse markdown to flat token stream. Tokens are markdown-it-py Tokens."""

class TokenRenderer(Protocol):
    """Sink-specific renderer. Each sink implements one of these."""
    name: str
    def render(self, tokens: list[Token]) -> str | list[dict]:
        """Return platform-native payload (string for telegram/email,
           list[dict] for feishu rich-text post)."""

def split_tokens(tokens: list[Token], max_size: int,
                 estimate_size: Callable[[list[Token]], int]) -> list[list[Token]]:
    """Split a token stream into sublists each estimated under max_size.
       Splits only at top-level block boundaries (paragraph, heading, list,
       code_block); never inside a code block, link, or list item.
       Operates on the AST, NOT the rendered output — see Feishu note below."""
```

**Token-aware splitter:** a key piece. Sinks need to fit messages under
platform limits (Telegram 4096 chars per message, Feishu **post** 30KB,
Email no hard limit ~100KB sane). Splitter walks tokens and emits chunk
boundaries between top-level blocks (paragraph, heading, list, code_block);
never splits inside a code block or hyperlink token group.

**Important constraint (per review 2026-04-29):** the splitter operates on
the **token stream**, not on rendered output. This means:
- Telegram (string output) and Feishu **post** mode (linear `list[dict]`):
  splitting works naturally — each chunk's tokens get rendered separately.
- Feishu **interactive (Card V2)** mode: cards are nested JSON trees, NOT
  linear payloads. Token-stream splitting cannot produce "two valid cards
  out of one" without re-templating. Therefore in Card V2 mode the
  splitter is **disabled**: oversized content goes through `truncate` or
  `fail` per `on_message_too_long`, never `split_more`. This restriction
  is enforced in `FeishuSink._validate_config()`.

**Renderers** live with their respective sinks
(`sluice/sinks/_telegram_render.py`, `_feishu_render.py`, `_email_render.py`),
not in the AST module — so AST stays sink-agnostic.

**Tests:** parser smoke (`# h1\n\nbody` → expected token kinds), splitter
correctness (input that would split inside a code block must split before),
each renderer tested against a fixture markdown corpus.

### Item 11 — Telegram sink

**Goal:** Push daily digest to a Telegram channel/group via Bot API.
Per-item split, daily_brief as first message, every message footers with
pipeline+date marker.

**Config:**
```toml
[[sinks]]
id            = "tg_main"
type          = "telegram"
bot_token     = "env:TG_BOT_TOKEN"
chat_id       = "env:TG_CHAT_ID"          # numeric or @channelname
brief_input   = "context.daily_brief"     # aggregate output dot-path; optional
items_input   = "items"                   # default: render each item via items_template
items_template = "prompts/tg_item.md"     # Jinja2 per-item template
split         = "per_item"                # "per_item" (brief first, then one msg per item)
                                          # | "single" (all in one message group, chunked at 4096)
# parse_mode is HARDCODED to MarkdownV2 — the AST renderer is MarkdownV2-
# specialized (escapes _*[]()~ etc.). Allowing user override would silently
# corrupt output. Not exposed as config (per review 2026-04-29).
link_preview_disabled = true              # Bot API 7.0+ link_preview_options.is_disabled
                                          # Default CHANGED from V1's implicit "preview on"
                                          # to "preview off" — daily-digest UX choice
                                          # (one big preview per item bloats the feed).
                                          # Set to false to restore previews.
footer_template = "📬 #{pipeline_id} {run_date}"   # appended to every message
                                                    # use empty string to disable
on_message_too_long = "split_more"        # "split_more" (split a single item across messages)
                                          # | "truncate" | "fail"
between_messages_delay = "1s"             # rate-limit politeness
```

**Behavior:**
1. Render `brief_input` via Jinja2 → markdown → AST → MarkdownV2 string;
   append footer; send as message 1.
2. For each item in `items_input`: render `items_template` → markdown → AST →
   MarkdownV2; append footer; send. Apply `between_messages_delay`.
3. If a single item exceeds 4096 chars:
   - `split_more`: token-aware split into N messages, append `(1/N)` etc to footer.
   - `truncate`: cut to fit, append `… [truncated]`.
   - `fail`: raise → sink fails → run fails.
4. State writes per send (also see §4 below):
   - One `sink_delivery_log` row per HTTP `sendMessage` call (success or
     failure; chunked single-mode counts each chunk separately).
   - On full-batch success, UPSERT one `sink_emissions` row holding
     `<chat_id>:<first_message_id>` as `external_id`.
5. **Push-channel idempotency model (telegram/feishu/email)** (revised
   per review 2026-04-29):
   - These sinks are **`create_new` by default and only**. Each pipeline
     run sends fresh messages. **No `mode` config key.** On Prefect retry
     within the same `run_key`, messages are re-sent (channel will see
     duplicates).
   - **Two state tables, two roles:**
     - `sink_emissions` (V1 schema): one row per
       `(pipeline_id, run_key, sink_id)`, holds the **latest** `external_id`
       (e.g. last sent message id). UPSERT semantics on retry — overwrites.
       This is "where to look up the most recent send for this run/sink",
       not an audit log.
     - `sink_delivery_log` (NEW V1.1 table, see §3): append-only, one row
       per **send attempt** (incl. retries, incl. per-recipient for
       email). This is the audit trail. Operators query via
       `sluice deliveries <pipeline_id> [--run RUN_KEY]`.
   - **Delivery log autocommit contract:** each `sink_delivery_log` INSERT
     is committed **immediately after each send attempt** (in its own
     transaction), NOT bundled into the V1 commit-after-success
     bucket. The point of the audit log is "an external side effect
     happened" — if the sink later raises and the run is rolled back, the
     external message was still delivered, and we MUST keep the audit
     row.
   - **SQLite single-writer constraint:** SQLite permits only one
     writer at a time. Two implementation rules follow:
     1. **Runner must NOT hold a long-lived write transaction across
        external sink sends.** The runner releases its write transaction
        (or runs sends outside any open write txn) before the sink-
        emission phase begins. Sink sends operate on their own connections.
     2. **Delivery log writes use retry + backoff** on `database is
        locked` (sqlite3 `OperationalError: database is locked`):
        100ms × 2^n exponential backoff, max 5 retries (≈3.1s total).
        On exhaustion, the audit row is dropped with a `log.error` entry
        — the actual send already happened, so dropping audit is a
        recoverability loss, not a correctness loss.
   - `sink_emissions` UPSERT remains in the normal commit-after-success
     path (it's about "where to find the latest external_id once the run
     is durable").
   - The earlier "audit only" wording on `sink_emissions` was wrong;
     `sink_emissions` cannot audit because its PK precludes multiple rows
     per run — `sink_delivery_log` does that job.
   - To suppress retries entirely, configure Prefect retry count = 0 on
     the pipeline.
   - Owner decision: "telegram 直接发啊？create 和 update 干啥呢？这个不
     是每天跑一次嘛" — accepted. Daily cadence + occasional retry-induced
     dupe is fine.

**`split = "single"`** behavior: render brief + items into one markdown,
then split-for-size at top-level block boundaries; all chunks treated as
one logical message group; `external_id` is the first chunk's id. Footer
is appended to the **last** chunk only (so multi-chunk single-split looks
like one logical message); in `per_item` split footer is appended per
item-message + per brief-message.

**MarkdownV2 escaping:** the renderer escapes the full set
(`_*[]()~\`>#+-=|{}.!`) outside of code/pre tokens; this is the standard
gotcha for Telegram and the AST approach makes it correct by construction.

**Bot API call** uses the modern `link_preview_options` object form per
Bot API 7.0+ (Mar 2024). The deprecated boolean `disable_web_page_preview`
is NOT used (some Bot API gateways already reject it).

**Tests:** unit (renderer escaping, splitter on a giant fixture markdown,
link_preview_options object shape); integration (httpx mock sending →
asserts message bodies, asserts modern API field used); duplicate-on-retry
contract (run twice in same `run_key` → asserts **two** send batches —
this is the create_new model, NOT a regression).

```toml
brief_input     = "context.daily_brief"
items_input     = "items"
items_template  = "prompts/email_item.md"
split           = "single"                   # default: one email per run
                                             #   "single": brief + all items in one email
                                             #   "per_item": separate email per item (rare; used
                                             #               for selective alerting)
html_template   = "prompts/email_wrapper.html.j2"   # outer HTML wrapper; receives
                                                     # {{ body_html }}, {{ footer }},
                                                     # {{ pipeline_id }}, {{ run_date }},
                                                     # {{ style_block }} (the contents of
                                                     # style_block_file, ready to drop in)
style_block_file = "prompts/email.css"       # optional; injected into <style>...</style>
                                             # in the wrapper. NOT true inline CSS — see
                                             # "Inline CSS naming" note below.
footer_template = "—\n📬 {pipeline_id} · {run_date} · sent by sluice"
attach_run_log  = false                      # debug aid: attach run summary as text
recipient_failure_policy = "fail_fast"       # "fail_fast" | "best_effort"
                                             #   fail_fast (default): any recipient
                                             #     SMTP failure → sink fails → run fails
                                             #     → seen_items NOT committed → retry
                                             #   best_effort: log per-recipient failures
                                             #     into sink_delivery_log (one row per
                                             #     recipient with status="failed"); sink
                                             #     succeeds if AT LEAST ONE recipient
                                             #     accepted. Suitable for newsletters with
                                             #     known-flaky addresses.
```

**Behavior:**
1. Markdown → AST → HTML via `_email_render.py` (uses markdown-it-py's
   built-in HTML renderer — emails are the one platform that *is* HTML).
2. Wrap in `html_template` with `style_block_file` contents injected at
   `{{ style_block }}`.
3. SMTP via `aiosmtplib` (true async; new dep — added to `[email]` extras
   and `pyproject.toml`).
4. One email message per `recipients` entry (BCC NOT used by default — we
   do not want recipients to see each other; explicit per-message send).
5. State writes per send (per Item 11 §4 model):
   - One `sink_delivery_log` row per recipient (success or failure;
     `recipient` column populated).
   - On full-batch success (or best_effort with ≥1 success), UPSERT
     one `sink_emissions` row with the first successful recipient's
     `<smtp_message_id>` as `external_id`.

**Inline CSS naming (per review 2026-04-29):** The config key was renamed
`inline_css → style_block_file` because we inject the CSS into a
`<style>...</style>` block, NOT into per-element `style="..."` attributes
("inline CSS" properly refers to the latter). True per-element inlining
would require a library like `premailer` or `css_inline`; this is NOT in
V1.1 scope. For Outlook compatibility, operators can either:
- Use only `<style>`-compatible CSS (works in Gmail, Apple Mail, modern
  Outlook); document known-broken Outlook quirks in template comments.
- Add `premailer` as a future feature (V2) that inlines styles before send.

**File:// attachment guard (per review 2026-04-29):** at config-load time,
if any pipeline has BOTH an Email sink AND a `mirror_attachments` stage
with `attachment_url_prefix` starting with `file://`, raise
`ConfigError("email sink requires attachment_url_prefix to be http(s):// —
local file paths cannot be loaded by email clients. Either configure a
public HTTP host for attachments, or remove mirror_attachments from
pipelines that emit to email.")`. CID-embedded inline images deferred
to V2.

**Recipient list note:** TOML static list per V1.1 owner decision. Future
V2 can add a SQLite `subscribers` table + `sluice subscribers add/remove`
CLI without breaking this contract — sinks would gain an optional
`recipient_source = "toml" | "db"` flag.

**Edge cases:**
- `recipient_failure_policy = "fail_fast"` (default per review 2026-04-29
  to preserve commit-after-success semantics): any SMTP failure stops the
  sink. Retry will resend to ALL recipients (including those who already
  got the prior attempt) — accepted under create_new-on-every-run model.
- `recipient_failure_policy = "best_effort"`: per-recipient failure
  recorded as a `sink_delivery_log` row with `status="failed"`,
  `recipient="<email>"`, plus error fields. Visible via
  `sluice deliveries <pipeline_id> --status failed`. Sink overall
  succeeds (run mark success) if at least one recipient delivered.
  These failures are **NOT** routed through `failed_items` /
  `sluice failures --retry` — that path is reserved for item-level
  dead-letter (which carries `item_json` and is re-injected as a pipeline
  Item on retry; recipient-level email failures don't fit that contract).
  To re-attempt a failed recipient send, the operator runs the pipeline
  again (creates a new `run_key` if cron has advanced; otherwise
  `sluice run <pipeline_id> --force-rerun` produces a new run that
  re-sends to all recipients).
- DKIM / SPF: out of scope; operator's responsibility to configure their
  SMTP relay.
- HTML email sanitization: not needed since input is our own rendered
  markdown; no user-controlled HTML enters the pipeline at this stage.

**Tests:** renderer fixtures, aiosmtplib mock send (single recipient
success path; fail_fast mode with one recipient failing → sink raises;
best_effort mode with one failing → sink succeeds + `sink_delivery_log`
row written with `status="failed"` and recipient populated), template
rendering with missing optional vars, config-validation (email + file://
→ ConfigError at load time).

## 5. Registry additions

```python
# sluice/registry.py — add:
register_enricher / get_enricher    # for Enricher protocol

# Plugin auto-registration model (per review 2026-04-29):
# Optional-dep modules MUST be lazily imported. Top-level package
# __init__.py files do NOT eagerly import telegram/feishu/email/etc.
# Instead, the registry uses STUB ENTRIES that import the real module on
# first construction:
#
#   register_sink("telegram", lazy="sluice.sinks.telegram:TelegramSink")
#   register_sink("feishu",   lazy="sluice.sinks.feishu:FeishuSink")
#   register_sink("email",    lazy="sluice.sinks.email:EmailSink")
#   register_processor("mirror_attachments",
#                      lazy="sluice.processors.mirror_attachments:...")
#   register_enricher("hn_comments",
#                     lazy="sluice.enrichers.hn_comments:...")
#
# These stubs are registered at sluice import time WITHOUT importing the
# heavy modules. The real module loads only when a TOML actually configures
# that sink/processor/enricher. If extras are missing at that point, the
# resulting ImportError is caught and re-raised as:
#
#   PluginExtraMissingError(
#     "Plugin 'telegram' requires `pip install sluice[telegram]` — "
#     "missing module: <missing>"
#   )
#
# Result: `sluice run` for a pipeline that does NOT use telegram works
# fine even when `aiosmtplib`/`markdown-it-py`/etc. are not installed.
```

## 9. Testing strategy

Same shape as V1: unit + integration + contract + idempotency. Coverage
target unchanged at 80%+. New test fixtures:
- `tests/fixtures/hn/sample_thread.html` — frozen hckrnws.com sample
- `tests/fixtures/markdown/` — markdown corpus for AST renderers
- `tests/fixtures/rss/with_full_content.xml` — for Item 7 fallback path

Mandatory new tests:
- Telegram / Feishu / Email sinks: run twice in same `run_key` → **two**
  send batches (create_new model is intentional — assert no idempotency
  suppression). This is the V1 `create_new`-mode pattern, exempted from
  no-duplicates idempotency tests by the V1 spec L668.
- Telegram sink with retry-after-partial-failure: assert duplicates show up
  on the channel (documented behavior, not a bug).
- `mirror_attachments`: run twice → `last_referenced_at` updated to the
  later run's timestamp; single file on disk selected by the url_hash
  winner (later runs reuse the winner's `local_path`, never overwrite
  with their own bytes — see §4.3 atomic write contract).
- `sink_delivery_log`: run twice in same `run_key` → at least 2 rows per
  send call (one per attempt); query `WHERE pipeline_id=? AND run_key=?`
  returns the full audit trail. `sink_emissions` row is the LAST send's
  external_id (UPSERT), unique per `(pipeline_id, run_key, sink_id)`.

## 11. Decisions locked

- `enrich` is a new processor type (`type = "enrich"`), not a new top-level
  protocol. Reuses Processor flow.
- AST library: `markdown-it-py` (not `mistune`, not custom). Reason:
  CommonMark conformance + active maintenance.
- Email recipients: TOML static list, no DB table in V1.1.
- `limit` ships with `group_by` + `per_group_max` from day one (owner
  decision: not deferred).
- `on_all_failed = "use_raw_summary"` fallback uses raw_summary as-is, no
  min-chars revalidation.
- `gc` does not delete from `run_log` (kept for stats / metrics).
- Sub-daily pipelines: `run_key_template` is a string with `{...}` placeholders,
  not a Python format spec, to avoid arbitrary code execution risk.
- Push-channel sinks (telegram/feishu/email) are `create_new`-only, no
  `mode` config key, no edit/upsert path. `sink_emissions` holds the
  LATEST send's external_id (UPSERT semantics). `sink_delivery_log`
  (NEW V1.1 table) is the append-only audit trail for every send attempt
  (including retries and per-recipient for email).
- Email `recipient_failure_policy = "best_effort"` writes per-recipient
  failures to `sink_delivery_log`, NOT to `failed_items` (the latter is
  reserved for item-level dead-letter that carries `item_json`).
- `notionify` v3 handles all image upload paths; sluice does not call
  Notion file-upload APIs directly.
- `attachment_mirror` eviction is by `last_referenced_at` only (no
  ref_count). Each cache hit bumps the timestamp; gc deletes rows
  unreferenced longer than `attachment_unreferenced_after`.
- `run_key_template` validation rule: rendered `run_key` for two
  consecutive cron fire times must differ. No string-pattern check on
  template variable names.
- Telegram parse_mode is hardcoded MarkdownV2 (not user-configurable).
  Bot API call uses `link_preview_options` object form (Bot API 7.0+),
  NOT the deprecated `disable_web_page_preview` boolean.
- Feishu `interactive` (Card V2) mode forbids `on_message_too_long =
  "split_more"` (caught at config load); `post` mode is the only one
  that supports auto-splitting.
- Email config key is `style_block_file` (NOT `inline_css`) since it
  injects into a `<style>` block, not per-element style attributes.
- Email `recipient_failure_policy` defaults to `fail_fast` to preserve
  commit-after-success semantics; `best_effort` is opt-in and writes
  per-recipient failures to `sink_delivery_log` (NOT `failed_items` —
  the latter is reserved for item-level dead-letter that carries
  `item_json`).
- Email + `attachment_url_prefix=file://` is a load-time `ConfigError`.
- All optional-dep plugins (telegram/feishu/email/metrics/hn_comments)
  use lazy registry stubs so `pip install sluice` (no extras) works for
  pipelines that don't use them.
- Item 3 `mirror_attachments` operates on items only (rewrites
  `item.attachments` and `rewrite_fields` dot-paths). Does NOT
  post-process rendered markdown.
- hn_comments uses a per-host token bucket shared across stage workers,
  not per-worker timer; default `request_delay_seconds = 10`.
- hckrnws.com / robots.txt: best-effort politeness only, no robots.txt
  parsing in V1.1.
